# dev-docs-pilot

A documentation Q&A agent over the Anthropic developer docs (`platform.claude.com/docs`). Built as a qualification project for the Cloud Employee **Senior AI Builder** role.

The repo is a monorepo:

- **`./` (root)** — TanStack Start + React 19 + Tailwind frontend (generated by Lovable, then iterated on).
- **`api/`** — FastAPI backend with a Claude Managed Agent (Claude Agent SDK), pgvector retrieval, LLM-as-judge reranker, and an evaluation harness.

The frontend speaks to the backend via SSE (`POST /chat`). The backend agent calls Anthropic + OpenAI (embeddings) + Supabase (pgvector).

## TL;DR architecture

```
Browser (Lovable / TanStack Start)
   │  POST /chat  (SSE: meta, tool_use, token, done, error)
   ▼
FastAPI  ───►  AgentRunner (Claude Agent SDK, claude-sonnet-4-6)
                    │
                    ├── tool: search_knowledge_base
                    │     ├── OpenAI text-embedding-3-small
                    │     ├── pgvector match_documents RPC (Supabase)
                    │     └── LLM-as-judge rerank (claude-haiku-4-5)
                    │
                    └── tool: format_citations
                          └── deterministic Sources block
   │
   ▼
Streaming markdown answer with inline + structured citations
```

## Quick start

### 1. Backend

```bash
cd api
cp .env.example .env
# Fill in: ANTHROPIC_API_KEY, OPENAI_API_KEY, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY
uv sync

# Schema is already applied to the assessment's Supabase project. To bootstrap a fresh
# project, run api/supabase/schema.sql in the Supabase SQL editor.

# Ingest the docs (uses /llms.txt convention — see api/README.md)
uv run docs-agent-ingest

# Start the API
uv run uvicorn docs_agent.api.main:app --reload
# → http://localhost:8000
```

### 2. Frontend

```bash
# From the repo root
bun install      # or: npm install
bun run dev      # → http://localhost:5173 (Vite/TanStack Start)
```

The frontend hits `http://localhost:8000/chat` by default. The Settings sheet (gear icon) lets you repoint to a deployed backend (e.g. an ngrok tunnel or Fly/Render deployment).

### 3. Eval

```bash
cd api
uv run python -m eval.run_eval
# Writes eval/runs/<timestamp>/report.md and persists aggregates to Supabase eval_runs
```

## Design highlights (a few worth pointing out)

### Why `/llms.txt` ingestion?

`docs.anthropic.com` is a JS-rendered SPA; a pure-httpx crawler gets back `"Loading..."` shells. The fix is the `/llms.txt` convention: many docs sites (Anthropic, Vercel, Stripe, Cloudflare) publish a markdown-friendly index of `.md` URLs. We use that as the default ingest mode; the BFS HTML crawler is the fallback (`--mode crawl`).

### LLM-as-judge reranker

Cosine similarity is a *relatedness* score, not a *sufficiency* score. A small judge model (Haiku, ~$0.0001/call, ~300ms) does real semantic discrimination on the top-N vector hits. Skips entirely when the top hit is already above `KB_HIGH_CONFIDENCE_SIM` (0.78) — fires on the murky middle, which is where retrieval failures cluster.

### Per-axis judge scores

The eval scores four axes independently — **faithfulness**, **completeness**, **citation correctness**, **hallucination** (binary, strict). Aggregate "quality" scores hide the failure mode. Per-axis lets us point at the cause and map it back to the anti-hallucination rules (AH-1..AH-6) in the system prompt.

### Anti-hallucination prompting (AH-1..AH-6)

Evolved from running RAG in production at MAVRYX Assistant. Most important: **never list what you think the docs cover, only what retrieval actually returned**. Without that rule, models confidently fabricate plausible-sounding feature lists.

## Eval

Latest run — `anthropic-docs-v2` dataset, 15 questions, agent `claude-sonnet-4-6`,
judge `claude-haiku-4-5` (2026-05-13). Reproduce with `cd api && uv run python -m eval.run_eval`;
results persist to Supabase `eval_runs` / `eval_results` and are browsable live at the
in-app `/evals` dashboard.

| Metric | Score |
|---|---|
| Precision@5 (retrieval) | 0.360 |
| Recall@5 (retrieval) | 0.600 |
| MRR (retrieval) | 0.593 |
| Faithfulness (judge) | 0.735 |
| Completeness (judge) | 0.913 |
| Citation correctness | 0.629 |
| Hallucination rate | 1.000 |

**Reading these honestly.** Completeness is high (0.91) — answers cover what's asked.
Retrieval is the weak axis: Precision@5 0.36 / Recall@5 0.60 means the right doc is
usually *found* but sits among noise, which drags citation correctness down to 0.63.
The 1.00 hallucination rate is the judge being deliberately strict — it flags *any*
detail not verbatim in a thin ground-truth set, even when that detail came straight
from a retrieved chunk. The `/evals` dashboard shows the per-question rationale so the
failure mode is legible rather than hidden behind an aggregate. Top of the roadmap:
tighten retrieval precision (better chunking + a stronger rerank threshold) and
calibrate the judge against a richer ground truth.
